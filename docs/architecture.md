# Архитектура MVP: 2D RPG для Telegram Web Apps

Цель: изометрический «вайб» Diablo 2 на ортогональной сетке с минимальным боем и лутом. Клиент — **mobile-first** (тапы + опционально стрелки на десктопе). Сервер — **Node.js** для будущей авторитативности; на этапе MVP допустим **клиент-авторитет** с валидацией там, где это дёшево.

## Стек и роли

| Слой | Технология | Роль |
|------|------------|------|
| Рендер | [PixiJS v8](https://pixijs.com/) (последний стабильный `pixi.js` на npm) | WebGL / WebGPU, `Application`, Assets, сцена, интерактив |
| Логика сущностей | [bitECS](https://github.com/NateTheGreatt/bitECS) | ECS: позиции, HP, теги, запросы в системах |
| Карта | [**pixi-tiledmap**](https://www.npmjs.com/package/pixi-tiledmap) | Загрузка `.tmj` / Tiled JSON под Pixi v8, слои → display tree |
| Ассеты MVP | `assets/` | Урезанный набор: одна карта, тайлсеты по ссылкам из `.tmj` |
| Полный пак | `full-assets/` | Источник для расширения; не тащить в бандл целиком |

## Три слоя: симуляция → представление → рендер

Сейчас в коде часто звучит «ECS и Pixi», но для роста проекта полезно зафиксировать **третий** контур — **presentation mapping** — чтобы не раздувать один **`RenderSync`** до помойки.

```text
Simulation (ECS + игровые системы + playerState для сумки)
        ↓  только чтение симуляции
Presentation mapping (реестр, синхронизация «что показать», VFX/камера как производные)
        ↓  обновление DisplayObject
Render (Pixi: сцена, слои, ассеты)
```

| Слой | Содержимое (ориентир) | Запрещено / граница |
|------|------------------------|---------------------|
| **Simulation** | **bitECS**: позиции, HP, бой, лут в мире, **`Animation`** как игровой компонент клипа, очереди **`DamageEvent` / `LootGranted`**, …; **`playerState.inventory`** — канон сумки | ❌ Импорт типов Pixi, ссылки на **`DisplayObject`**, решение «как красиво» вместо «что истинно в мире» |
| **Presentation** | **Реестр** (`renderId` → узел), **`RenderSyncSystem`**: ECS → позиция спрайта, кадр анимации, alpha по таймеру лута, видимость. **Дальше сюда же** (отдельные модули / системы, не ECS): отложенные **VFX** (вспышка удара), **сглаживание камеры** (lerp к цели из симуляции), **экранные тряски** — всё **производно** от симуляции, **без** записи в **`Health` / `Position`** ради «эффекта» | ❌ Мутация симуляции ради картинки (см. контракт **`RenderAdapter`** для обратного направления). ❌ Пихать бесконечно логику в один файл **`RenderSync`** — см. риски ниже |
| **Render** | **Pixi**: `Application`, контейнеры, **pixi-tiledmap**, загрузка текстур | ❌ Правила боя и инвентаря |

**Обратный поток (Pixi → игра):** только **`RenderAdapter` → `RenderEvent[]` →** симуляция в зафиксированных фазах — без подписок на **`sprite.on`** из игровых систем.

**Зачем явно:** когда появятся **анимационные эффекты**, **VFX**, **состояния HUD**, **камера** (shake, zoom, cinematic), их естественно вешать на **presentation** (отдельные проходы после симуляции, перед или вместе с обновлением реестра). Если называть всё это «просто RenderSync», слой слипнется: смешаются **истина мира** и **визуальный сахар**, сложнее тестировать и вынести сервер (на сервере **Render** и часть **Presentation** нет, **Simulation** остаётся).

## Связка ECS ↔ рендер (адаптер, не жёсткий 1:1)

**Правило:** слой симуляции (**ECS**) **не импортирует** типы Pixi и не держит ссылки на `DisplayObject`. Вместо «один `eid` = один спрайт навсегда» используется косвенная связь через **presentation** (реестр).

**Компонент (пример):** `RenderRef { renderId }` — непрозрачный идентификатор (число / UUID) записи в **реестре представлений** снаружи ECS (`Map<renderId, Container | Graphics>` или пул узлов). Один `eid` может иметь несколько `RenderRef` (тень + тело + оружие), либо несколько сущностей с общим визуальным корнем — на выбор команды; важно, что ECS знает только `renderId`.

**Разделение ответственности (чтобы не «протыкать» ECS из Pixi):**

1. **`RenderSyncSystem`** — часть **presentation**: **синхронизация симуляция → реестр → Pixi** (читает `Position`, `Health`, видимость, **`Animation`**, …; обновляет узлы по `renderId`). **Не** создаёт и **не** удаляет **сущности ECS** — иначе скрытые сайд-эффекты и рассинхрон. Движение и бой **не** трогают `sprite` напрямую. Новые чисто-визуальные фичи (**VFX**, **камера**) — **отдельные** модули presentation, не обязательно внутри этого класса.

2. **`RenderAdapter` (мост Pixi → симуляция)** — **единственная точка**, где события от **рендера** ставятся в очередь; игровой слой их **не подписывает напрямую** на `sprite.on(...)`.

### Контракт RenderAdapter (формальный)

**Жёсткое правило (must-have на весь срок проекта):** код **`RenderAdapter`** и всех колбэков, которые он регистрирует на Pixi:

1. **ECS — только чтение** (если обращение к миру вообще нужно: проверка `eid`, запрос компонента **без записи**). Часто достаточно **`entity`** и координат из замыкания — без доступа к ECS в адаптере.
2. **Наружу в игру — только события:** пополнение **внутренней очереди** `RenderEvent` и выдача через **`poll()`**. Никаких побочных эффектов на игровом состоянии вне этой очереди.
3. ❌ **Никакой мутации ECS** из адаптера: не **`addComponent` / `removeComponent`**, не **`addEntity` / `removeEntity`**, не запись в **`Position`**, **`Health`**, **`Animation`**, инвентарь и т.д.

Иначе колбэки рендера превращаются во **второй, неупорядоченный игровой цикл** — гонки с тиком, неповторимые баги и невозможность вынести логику на сервер. Менять мир могут **только** системы после **`poll()`**, по зафиксированным фазам.

Типы событий (имена и поля можно слегка расширить, но принцип — **discriminated union** + один вход в ECS):

```ts
// entity — bitECS eid (number)
// lane: опционально; по умолчанию вести как gameplay (обработка в тике N+1).
type RenderEvent =
  | { type: 'ANIMATION_COMPLETE'; entity: number; lane?: 'gameplay' | 'cosmetic' }
  | { type: 'POINTER_TAP'; worldX: number; worldY: number; lane?: 'gameplay' | 'cosmetic' }
  | { type: 'SPRITE_READY'; entity: number; lane?: 'gameplay' | 'cosmetic' }

interface RenderAdapter {
  /** Снять все события, накопленные с прошлого кадра (или с момента последнего poll). */
  poll(): RenderEvent[]
}
```

**Правило (деталь к жёсткому выше):** обработчики Pixi (`complete`, `pointertap`, `texture` loaded, …) **только** пишут во **внутреннюю очередь** адаптера. **ECS-системы и остальной игровой код** читают рендер **только** через `poll()` (момент вызова — по **фазам тика** ниже: конец тика и/или начало следующего — зафиксировать в коде).

**Лут и VFX:** **`RenderAdapter` не должен определять судьбу сущности** (удаление лута, выдача в инвентарь). Подбор → состояние + **игровой таймер** на сущности → **RenderSync** рисует fade; по обнулению таймера **игровая система** удаляет entity. См. [post-mvp-development-plan.md](./post-mvp-development-plan.md), фаза 2 (Loot VFX).

Соответствие примерам:

| Событие Pixi | `RenderEvent` |
|--------------|----------------|
| конец клипа анимации | `ANIMATION_COMPLETE` |
| тап по `DisplayObject` (мир уже переведён в `worldX` / `worldY`) | `POINTER_TAP` → дальше тот же слой, что **`PlayerIntent`** / hit-test |
| текстура/спрайт готов, известны размеры кадра | `SPRITE_READY` → система обновляет **`Hitbox`** и т.д. |

**Импорт и мутация `DisplayObject` для игровых сущностей** остаются в коде рендера (sync + адаптер + фабрика узлов реестра), но **семантически**: sync только «рисует состояние мира», adapter только «копит и отдаёт `RenderEvent[]`».

**Задержка в 1 тик (N → N+1):** для событий, влияющих на **FSM / логику смерти**, политика **gameplay** — consume в **N+1**. Для **чисто косметики** (ударный блеск, не связанный с `DamageEvent`) можно пометить **`lane: 'cosmetic'`** и обрабатывать **сразу** после `poll()` в конце кадра N, чтобы уменьшить ощущемую задержку — не смешивать с удалением entity и не нарушать единый поток без явной метки.

**Зачем:** несколько визуальных элементов на сущность, пересоздание спрайта без пересоздания `eid`, скрытие / замена вида без протекания API рендера в боевую логику — и без разрастания вызовов `addComponent` из обработчиков по всей сцене.

## Ввод: Input → Intent

Сырые события (тап, стрелки, pointer) **не** напрямую двигают игрока и **не** сразу наносят урон. Отдельный слой собирает кадр (или тик) в структуру **намерения**:

```ts
// Пример формы; поля опциональны по кадру
type PlayerIntent = {
  moveTo: { x: number; y: number } | null       // десктоп: куда идти (цель в мире)
  moveDirection: { x: number; y: number } | null // мобила: нормализованное направление от тапа
  attackTarget: number | null                    // eid цели или null (bitECS entity id)
}
```

Сборщик ввода выставляет **либо** целевую точку, **либо** направление (см. таблицу desktop/mobile и правило **остановки** движения на мобиле в [implementation-plan.md](./implementation-plan.md)). **Системы решения** (порядок фиксирован): проверка валидной атаки по `attackTarget`; иначе движение по `moveDirection` или к `moveTo`. Так проще добавлять `USE_ITEM`, приоритеты, станы, GUI без переписывания низкоуровневого ввода.

## Намерения анимации (рост после MVP)

**Gameplay state и animation state — не смешивать (долгая жизнь проекта):** строки **`idle` | `walk` | `attack` | `hurt` | `death`** относятся к **клипам / визуальному FSM**. Если по ним же судить о бое («в `attack` — значит активен хитбокс» или «в `hurt` — значит стан»), получится хрупкая связка логики и таймингов спрайта — сломается при смене ассета, сетевой задержке или правке **`interruptAt`**. ✅ Развести явно:

| Слой | Роль | Примеры значений (расширяются отдельно от клипов) |
|------|------|---------------------------------------------------|
| **Gameplay / combat state** (канон для логики; имя компонента условное: **`CombatState`**, **`UnitPhase`**, …) | Жив ли юнит, атакует ли **логически**, стан, неуязвимость после хита | **`alive`**, **`dead`**, **`attacking`**, **`stunned`**, … |
| **`Animation.state`** | Только **какой клип** играет и как FSM клипа переключается | **`idle`**, **`walk`**, **`attack`**, **`hurt`**, **`death`** |

**Правило производности:** **`Animation.state`** (и при необходимости внутренние **`AnimationRequest`**) выводятся из **gameplay state + движение + приоритеты клипов + `ANIMATION_COMPLETE`** — желательно **одним** слоем (**`AnimationSystem`** целиком или **`AnimationDriver`** → затем FSM). Бой, AI и ввод меняют **сначала** намерения боя и **`CombatState` / `Health`**, а не «лезут в спрайт» как в единственный источник правды. ❌ Не проверять **`stunned`** только по **`Animation.state === 'hurt'`**.

**Анимация не влияет на gameplay (must-have):** **`Animation` / FSM** в симуляции + **presentation** (**`RenderSync`**, обработка **`ANIMATION_COMPLETE`** через адаптер) — **только визуал** относительно боя. ❌ Не из них вызывать **`emitDamage`**, не менять **`Health`**, не выставлять **`AttackIntent`**, не переключать **`CombatState`** как единственный эффект кадра клипа. ❌ Не вешать бой на **`sprite.onFrame` / `currentFrame`**. Иначе порядок с **`CombatSystem`**, сервер и реплей разъедутся. Бой остаётся в конвейере **`AttackIntent` → … → `DamageEvent` → `Health`**.

**Fallback при неизвестном или сломанном состоянии:** зафиксировать **`defaultState = 'idle'`** (константа, напр. **`ANIMATION.DEFAULT_STATE`**). Если в **`AnimationRequest`** пришло неизвестное **`state`**, в **`AnimationClips`** **нет** записи для персонажа, или переход неконсистентен → перевести **`Animation.state`** в **`idle`**, сбросить/нормализовать **`time`** и играть безопасный клип; в **dev** — лог. Без этого сущность **зависает** в несуществующем состоянии после опечатки, бага данных или будущего расширения enum.

Бой, AI и ввод **не** меняют напрямую **`Animation.state`** у сущности как единственный контур — иначе FSM разъедут несколько авторов. В очередь на анимацию попадает **запрос** (идеально — после сведения gameplay, см. выше):

```ts
type AnimationRequest = {
  entity: number
  state: 'idle' | 'walk' | 'attack' | 'hurt' | 'death'
  force?: boolean // редко: пробить часть блокировок (использовать осознанно)
}
```

**`AnimationSystem` (FSM по клипам)** читает запросы и текущий **`Animation`**, решает переход по **приоритетам** визуальных состояний (`death > hurt > attack > walk > idle`). События из **`RenderAdapter`** (`ANIMATION_COMPLETE`) **не** пишут в `Animation` напрямую — переводятся в запросы/флаги в **одной** точке. Детали: **`minHoldTime`**, **`interruptAt`**, анти-дребезг — [post-mvp-development-plan.md](./post-mvp-development-plan.md), фаза 1.1.

**Несколько `AnimationRequest` за один тик на одну сущность:** ❌ не складывать в FIFO-очередь без правила — порядок вызовов систем даст **недетерминизм** (например **attack** от игрока и **hurt** от врага в одном кадре). ✅ За тик на **`eid`** держать **ровно один** агрегированный запрос: **`AnimationIntentBuffer[eid] = bestByPriority(requests)`**, числовые приоритеты состояний (выше = жёстче): **`death: 100`**, **`hurt: 80`**, **`attack: 60`**, **`walk: 20`**, **`idle: 0`**. При равном приоритете — **детерминированное** правило (например порядок источников в коде или **`seq`** на запросе). Уже **после** этого сравнивать с текущим **`Animation.state`**, **`locked`**, **`minHoldTime`**, **`interruptAt`**.

**Анти-spam (дополнительно):** если **`requestedState === currentAnimation.state`** без смысла перезапуска клипа — не обновлять буфер дубликатом. Иначе AI засыпает **`walk, walk, …`**.

**Направление (`Facing`) — не часть `Animation.state`:** держать отдельным компонентом (угол или `up | down | left | right`). Иначе FSM раздуется до `walk_up`, `walk_down`, … Вместо этого: **`state: walk`** + **`facing: left`** (зеркало спрайта в **presentation** / **`RenderSync`**). **`facingLocked` — когда снять:** явное правило, иначе персонаж «залипнет» разворотом после атаки. Варианты (выбрать один в коде и не смешивать): (1) по **анимации**: **`Animation.state === 'attack' && time / duration < interruptAt`**; (2) **`facingLocked := Animation.locked`**; для **геймплейных** проверок (можно ли сменить цель) опираться на **`CombatState`**, а не только на клип. Пока **`facingLocked`**, ввод **не** меняет **`Facing`**.

## Движение: желаемая и фактическая скорость

Чтобы **AI**, **ввод**, **knockback**, **slow** и **stun** не ломали друг друга, развести:

| Компонент / поле | Кто пишет | Смысл |
|------------------|-----------|--------|
| **`DesiredVelocity`** | **Один** проход / **одна** система намерения движения (например **`MovementIntentSystem`**) | Итог «куда хочет идти сущность» после агрегации **ввода**, **`AIIntent`**, станнов (обнуление). ❌ **Нельзя**, чтобы **AI**, **ввод** и **knockback** **независимо** писали в одно поле — последний перетрёт остальных. |
| **`ExternalForces` / `ImpulseBuffer`** (имя условное) | Бой, **knockback**, отталкивания | Вклады, которые **не** являются «намерением идти»; суммируются в **`MovementSystem`** при расчёте фактической скорости. |
| **`ActualVelocity`** (факт за кадр) | **Movement + Collision** (один слой) | После **`DesiredVelocity` + внешних сил**, коллизий, модификаторов. |

Интеграция позиции: **`Position += ActualVelocity * dt`** (или эквивалент с разрешением коллизий по шагам). **Инвариант:** **`Position` напрямую меняет только `MovementSystem`** (или один согласованный модуль «физика мира»), который учитывает **`ActualVelocity`**, коллизии, телепорты уровня дизайна. ❌ **Knockback не через перезапись `DesiredVelocity` у AI** — иначе AI и отбрасывание будут бороться за одно поле. Подробнее — [post-mvp-development-plan.md](./post-mvp-development-plan.md), фаза 3.

**Единственный продьюсер `DamageEvent` (must-have):** ❌ **AI**, **ввод**, ловушки, DoT и любые системы **вне** боевого конвейера **не** вызывают **`emitDamage`** и **не** пушат **`DamageEvent`**: они выставляют **`AttackIntent`** (и при необходимости сигналы зон/тиков DoT). В **`queues.current.damage`** попадает только **один** согласованный пайплайн фазы **2** — иначе снова **двойной урон** / **пропуск** при смене порядка систем.

**Внутри конвейера — не монолит (`CombatSystem` ≠ God Object):** если всё свалить в один класс «и валидация, и формула, и сервер на будущее», при добавлении **крита, брони, баффов** он раздуется и сломается. ✅ Разделить **решение** и **результат** (имена условные, порядок фиксирован в фазе **2**):

```text
AttackIntent
  → TargetResolutionSystem   // кто цель, дистанция, кулдаун, LOS, жива ли цель — «можно ли бить»
  → DamageCalculationSystem  // база, модификаторы, крит, броня, баффы — «сколько`
  → DamageEventEmitter       // только сборка DamageEvent + eventId + emitDamage(...)
```

**`CombatSystem`** — **оркестратор**: вызывает этапы **в одном порядке** (или это три системы подряд без других продьюсеров между ними). **`emitDamage`** вызывается **только** из **`DamageEventEmitter`** (или одной приватной функции с тем же смыслом). **AI:** **`AIIntent` → `AttackIntent` → … → `DamageEvent`**, как у игрока.

## Клиент: слои приложения

1. **Bootstrap TWA** — `Telegram.WebApp.ready()`, размер viewport, отключение зума страницы при необходимости, CSS `100dvh` / safe-area.
2. **Asset pipeline** — расширения **pixi-tiledmap** для `Assets` в Pixi v8; файлы под **`public/assets/`**, в `.tmj` и тайлсетах — пути **с префиксом `/assets/`** и `fetch('/assets/map.tmj')` без кастомного basePath (см. [implementation-plan.md](./implementation-plan.md), фаза 2.1).
3. **Карта** — через **pixi-tiledmap**: визуальные слои `ground`, `wall`; слой `collisions` скрыть (в Tiled или `visible = false` на контейнере слоя). Массив `data` слоя `collisions` читать для **физики** (или экспортировать отдельный буфер при загрузке).
4. **ECS-мир** — отдельный объект `gameWorld` (не путать с `createWorld` из bitECS — можно назвать `ecsWorld`). Компоненты в SoA: **`Position { x, y }` в мировых пикселях** (как Tiled), `Velocity`, `Health`, `Hitbox`, флаги `Player`, `Enemy`, `Loot`, `Dead` и т.д. Тайлы — только через `floor` при проверке сетки (см. [implementation-plan.md](./implementation-plan.md)). Рекомендуемые **теги** (или отдельные tag-компоненты): **`TagPlayer`**, **`TagEnemy`**, **`TagLoot`** — дешёвые фильтры в системах и отладка.
5. **Игровой цикл (тик)** — один **зафиксированный порядок фаз**; время из тикера (`deltaMS` → секунды, кап — см. [implementation-plan.md](./implementation-plan.md)).

### Фазы тика (tick order)

| # | Фаза | Смысл |
|---|------|--------|
| 1 | **Input → Intent** | Сырой ввод → **`PlayerIntent`** (`moveTo`, `moveDirection`, `attackTarget`). |
| 2 | **Intent → Events** | Конвейер боя (**оркестратор + resolution + расчёт + `DamageEventEmitter`**) — **единственный** источник **`DamageEvent`**; прочие намерения → свои события. Очередь **`LootGranted`** и т.д. — свои продьюсеры. Не смешивать с немедленным списанием HP в том же проходе. |
| 3 | **Events → State changes** | Применение событий: **`Health`**, инвентарь, смерть → лут, таймеры VFX лута, кулдауны. |
| 4 | **Movement + Collision** | Смещение позиций, коллизии с сеткой (игрок и прочие, кому положено в этой фазе). |
| 5 | **AI** | Логика врагов **после** обновления намерений/событий/движения за кадр (преследование, атака, stuck). При большом числе NPC — **фазировать `think()`** по **`tickId`** (**`thinkOffset`**, группы) — см. [post-mvp-development-plan.md](./post-mvp-development-plan.md), фаза 3. |
| 6 | **Presentation → реестр / Pixi** | В первую очередь **`RenderSync`**: симуляция → реестр по **`RenderRef`**, **`Position`**, **`Animation`**, … — без мутации ECS. Сюда же по мере роста — отдельные проходы **VFX / камера** (presentation), если они не раздувают один класс. |
| 7 | **`RenderAdapter.poll()`** | Снять **`RenderEvent[]`** из очереди адаптера (граница **Render → Simulation**). |

**Зачем так:** предсказуемость, меньше скрытых гонок, проще вставлять системы — всегда видно, до или после движения / AI они выполняются.

**`GameEventQueues` и этот порядок:** в кадре **N** фаза **2** пишет только в **`current`**; фаза **3** читает **`processing`** (содержимое с кадра **N−1**, после **`swap()` в конце N−1**), затем **`processing.clear()`**. В **конце кадра N** (логично сразу после фазы **7** и обновления **`tickId`**) — **`swap()`**, чтобы накопленное в **`current`** стало **`processing`** для кадра **N+1**. Иначе фазы **2** и **3** в одном кадре конфликтуют с одним массивом. Детали — блок «Где живут события» ниже.

**Политика шага 7 (`poll()`) — зафиксировать явно:**

```text
Кадр N: фазы 1–6 → RenderAdapter.poll() → события попадают в буфер игры (очередь на следующий тик)

Кадр N+1: в начале тика (до или вместе с фазой 1–2) — consume / раздача накопленных RenderEvent
```

Так меньше **недетерминизма** и реже ситуация «`ANIMATION_COMPLETE` то срабатывает, то нет» из-за порядка колбэков внутри кадра. Если нужна обработка **в том же** кадре N — документируйте **другой** контракт (например второй `poll()` в начале N+1 не используется, а события сливаются сразу после фазы 6) и придерживайтесь его везде.

**Камера** — после симуляции, в **presentation** (фаза **6** или выделенный шаг сразу после **`RenderSync`**): цель берётся из **ECS** (**`Position`** игрока), а **lerp**, **deadzone**, будущие **shake / zoom** — в узлах Pixi или промежуточной «камерной» структуре, **без** записи сглаженной позиции в **`Position`** как канона мира — [post-mvp-development-plan.md](./post-mvp-development-plan.md).

**MVP (сопоставление):** смерть врага и спавн лута — в фазе **3** после применения урона; подбор лута — намерение во **2** (или 1), результат в **2–3**.

### Время (GameTime): один источник истины

Все системы берут время из **одного** контекста тика (имя условное):

```ts
type GameTime = {
  dt: number      // длительность тика в секундах (из Ticker / performance, с верхним капом)
  now: number     // монотонные часы в секундах (или ms — но тогда везде ms)
  tickId: number  // инкремент +1 каждый игровой тик (логи, события, см. DamageEvent)
}
```

**Правило:** не вводить параллельно «таймеры в миллисекундах» и «в секундах» в разных модулях. Примеры согласования с **`dt`** (в секундах):

- `animation.time += gameTime.dt`
- `pickupVfxTimer -= gameTime.dt`
- `attackCooldown -= gameTime.dt`

Иначе через несколько фаз появятся рассинхрон анимации и логики и ошибки масштаба времени.

## Коллизии (MVP)

Текущая карта (`assets/map.tmj`): ортогональная сетка **32×32**, слой **`collisions`** — тайлы с ненулевым gid трактуются как **непроходимые** (например gid `30` по всему периметру и препятствиям).

Алгоритм:

- **Игрок / враг / лут** — ось-выровненный **AABB** в **мировых пикселях** (квадрат в мире = один размер, например 24×24 с отступом от клетки).
- **Доступ к сетке:** `isBlockedTile(tx, ty)`; после смещения — проверка **четырёх углов** хитбокса → тайлы через `floor(world / tileSize)` (детали в плане имплементации).
- **Ввод:** один перевод **screen → world** с учётом камеры и `worldScale`, чтобы клики и `Position` были в одной системе.
- **Разрешение:** часто пробное смещение по X, затем по Y с откатом оси; в углах возможны застревание и дрожание — при необходимости сеточный шаг или мелкие подшаги (см. план имплементации).
- Альтернатива позже: object layer с полигонами — не нужна для MVP с квадратами.

## Бой и лут (MVP)

- **Атака:** тап → мировые координаты → в **`PlayerIntent`** выставить `attackTarget` или поля движения. Урон только если одновременно: цель **живая** (`Enemy`, не `Dead`), **дистанция \< attackRange**, **кулдаун** (порядка 300 ms) истёк — иначе без урона. **Не привязывать урон к кадру анимации** (см. ниже и [post-mvp-development-plan.md](./post-mvp-development-plan.md)).
- **Смерть:** `Health <= 0` → компонент `Dead`, отключение коллизии врага, удаление/скрытие спрайта → создать сущность лута в той же позиции.
- **Лут:** один тип (например «монета») — квадрат другого цвета; подбор при пересечении с игроком (или тап по луту — на мобиле удобнее авто-подбор при касании).

## Сервер Node.js (MVP и рост)

**MVP:** сервер раздаёт статику (Vite build / `dist`) и опционально JSON health. Игровое состояние полностью на клиенте (**клиент-авторитет** — допустимо для одиночного MVP).

**Заложить сразу (античит-мышление, даже без логики на сервере):**

- не принимать с клиента «голый» урон или смерть цели как истину; в проде урон считает сервер по своему `ATTACK` и таблицам;
- не принимать произвольные телепорты позиции; в проде `MOVE` валидируется по скорости, dt, коллизиям с миром сервера;
- клиент может **отображать** предикт, но канон — версия сервера (когда появится).

**Единый контракт событий игрока** — завести типы/enum и очередь на клиенте уже в MVP (хотя бы `console` или no-op отправитель), чтобы потом тот же payload уйти по WebSocket:

| Событие | Назначение (смысл для будущего сервера) |
|---------|----------------------------------------|
| `MOVE` | намерение сместиться (цель, направление или снапшот ввода — уточнить схему; сервер проверяет допустимость) |
| `ATTACK` | намерение ударить цель `targetId` (сервер проверяет дистанцию, кулдаун, LOS) |
| `USE_ITEM` | использование предмета слота (сервер проверяет наличие, кулдаун, контекст) |

Тела сообщений — отдельные TypeScript-типы в общем пакете `shared/` или `packages/protocol/`, чтобы клиент и Node использовали одни и те же имена полей.

**Внутренний пайплайн боя и лута на клиенте (готовность к авторитетному серверу):** по возможности **не** применять урон и смерть в той же системе, которая только «решила атаку». Развести этапы: намерение / валидация (**`ATTACK`**, **`PICKUP`**) → **событие результата** → система, которая **меняет `Health` / инвентарь / мир**.

**Где живут события:** явный слой очередей с **жёстким double-buffer** (must-have для фазы 5 и сервера — одинаковый порядок, нет случайного double-consume, удобный лог **`processing`**):

```ts
type GameEventQueues = {
  /** Писать только сюда в фазе 2 текущего кадра. */
  current: { damage: DamageEvent[]; loot: LootGranted[] /* … */ }
  /** Читать только отсюда в фазе 3 следующего кадра (после swap). */
  processing: { damage: DamageEvent[]; loot: LootGranted[] /* … */ }
}
// swap(): поменять current ↔ processing в конце кадра (после фазы 7 / перед инкрементом tickId)
```

**API без прямого доступа к массивам:** ❌ не экспортировать **`queues.current.damage.push`** наружу — фаза **3** или ошибка продьюсера могут **писать в `processing`** и сломать порядок. ✅ Обернуть в объект с методами вроде **`emitDamage(e)`** / **`emitLootGranted(…)`** (всегда пишут **только** в **`current`**) и **`getDamageEvents()`** / **`getLootGranted()`** для потребителей фазы **3** (возвращают **только** срез **`processing`**, read-only или копия до **`clear`**). Внутренняя структура — **приватная**. **`emitDamage`** из игрового кода вызывает **только** **`DamageEventEmitter`** (или эквивалент — одна точка внутри конвейера боя; см. правило выше; тесты — отдельно).

**Контракт по кадрам:**

```text
Кадр N:
  фаза 2 — продьюсеры пушат только в queues.current

Конец кадра N:
  queues.swap()   // то, что накопили в current, становится буфером для чтения в N+1

Кадр N+1:
  фаза 2 — новые события пишутся в queues.current (уже «чистый» после swap)
  фаза 3 — потребители читают только queues.processing, применяют изменения мира
  после фазы 3 — queues.processing.clear() на каждую внутреннюю очередь (damage, loot, …)
```

Не смешивать «писать и читать один массив в одном кадре» без такого разделения — иначе легко **дважды применить** событие или потерять порядок относительно будущего серверного тика.

Пример **`DamageEvent`**:

```ts
type DamageEvent = {
  tickId: number
  /** Обязательный uniqueId события в рамках сессии/мира (не опционально). */
  eventId: string // например `${tickId}-${monotonicSeq}` или uuid v4
  sourceType: 'entity' | 'environment' | 'dot'
  sourceId: number // при sourceType === 'entity' — eid; иначе зарезервированный id зоны/ловушки/тика DoT (явная схема, без «-1»)
  targetId: number // eid цели
  amount: number
  sourceX: number
  sourceY: number
}
```

**Обязательный контракт (идемпотентность):** один и тот же логический удар не должен **дважды** изменить **`Health`**, если очередь переигралась, **система применения упала и перезапустилась**, **фаза 3 вызвалась повторно по ошибке** или пришёл **дубликат с сервера/реплея**. Правило для **`HealthSystem`** (или единственного потребителя урона из **`processing`**):

1. Перед применением: если **`eventId`** уже есть в **`processedEvents`** (или эквивалент) → **игнорировать** событие (**no-op**), не менять HP.
2. После успешного применения: **записать** **`eventId`** в **`processedEvents`** (атомарно с применением в одном проходе по событию).

**`eventId`** — не декорация: без него **`tickId` одного** недостаточно (несколько попаданий в тике) и не защитит от повторной доставки того же события. Вместе с **единственным конвейером эмиссии** снижает риск **двойной эмиссии** при рефакторинге порядка систем.

- **Хранение `processedEvents`:** скользящее окно по тикам, чтобы **`Set` не рос бесконечно**, например **`Map<tickId, Set<eventId>>`**: при продвижении **`currentTick`** удалять записи с **`tickId < currentTick - N`** (**N ≈ 30–60**). Проверка дубликата — по **`eventId`** (в рамках выбранной схемы достаточно глобально уникального **`eventId`**, тогда один **`Set<eventId>`** + LRU тоже ок).
- **`sourceType`** — ловушка, зона, DoT и сущность различаются явно; **`sourceId`** + **`sourceX/Y`** согласованы с типом (для **`environment`** — якорь зоны в мире).
- **`sourceId` / `targetId`** при **`entity`** — агро, «кто убил», несколько источников за тик; логи и реплей.
- **`sourceX` / `sourceY`** — позиция источника в мировых пикселях (отбрасывание, VFX направления удара, попапы урона); для статических ловушек можно брать центр зоны или последнюю известную точку источника.

**Урон и анимация:** цепочка **`AttackIntent` → (конвейер боя: resolution → расчёт → emit) → `DamageEvent` → (фаза 3) `Health`**. Клип атаки — **только визуал**. ❌ Не наносить урон с **конкретного кадра спрайта** или `onFrame` — иначе авторитетный сервер и сетевая сверка сломают модель.

Тогда при появлении сервера каноничными становятся **события результата** с сервера (или подтверждённые дельты), а не перепись всего боя. Подробнее — [post-mvp-development-plan.md](./post-mvp-development-plan.md) (фаза 5 и принципы).

**Единственный источник правды (must-have):**

- **ECS** — канон для **мира**: сущности, **`Position`**, **`Health`**, лут **на земле**, враги, компоненты боя/движения и всё, что относится к симуляции карты.
- **`playerState`** — канон **только** для **инвентаря** (**`inventory`**) и, по мере роста, **чистой меты** (флаги квестов, настройки — **не** позиция и **не** HP). Сумка **не** дублируется как «истина» в ECS-слотах.

**Кэши и проекции (UI, React state, поля в системах «для удобства»):** допустимы **только как derived data** — их всегда можно заново вывести из **ECS + `playerState`**. ❌ Нельзя считать такой кэш **источником истины**: не принимать решения о мире или инвентаре только по значению из UI; не хранить там единственную копию HP/позиции/состава сумки. Иначе неизбежны десинк, «почему UI врёт» и регрессии после многих фич.

**Граница `playerState` и ECS (таблица):**

| Данные | Где живут |
|--------|-----------|
| `Position` игрока (и прочих существ в мире) | **ECS** |
| `Health` игрока (и врагов) | **ECS** |
| Инвентарь / сумка | **`playerState.inventory`** (не ECS) |
| UI (HUD, оверлеи) | **Читает** ECS + `playerState`; локальный state — только **отображение** / анимация / фокус, не канон |

**Правило:** **`playerState` не хранит данные мира** — не копировать в него позицию, HP, состояние врагов. Сейв: мир из ECS, мета и сумка из `playerState`.

**Инвентарь после MVP:** сущности **`Loot` в мире** остаются в ECS; **содержимое сумки** — в **`playerState`** (например `inventory: ItemStack[]`), не набор сущностей-слотов в ECS. **Лут `reserved`:** если **`reservedBy`** перестала существовать в мире → **`LootState = idle`** (обязательное правило, иначе залипание) — [post-mvp-development-plan.md](./post-mvp-development-plan.md), фаза 2.

**API инвентаря:** не мутировать массив инвентаря из множества систем напрямую. Один **транзакционный** вход для добавления: **`InventoryService.tryAddItem(…) → { ok, reason? }`** — внутри **вся бизнес-логика** (стак, лимит слотов, уникальные предметы, квестовые ограничения, вес — что появится) и **атомарно** «проверка + мутация» в одном вызове. При **`ok: false`** инвентарь **не** меняется; **`reason`** — машиночитаемый код (**`inventory_full`**, **`stack_cap`**, **`unique_owned`**, …) для UI, логов и **серверного отказа** в том же контракте. ❌ Не дублировать эти правила в **`LootSystem`** (нет отдельного «предварительного canAdd» с копией формул — иначе рассинхрон с сервером и двойное сопровождение). Плюс **`removeItem`**, **`hasItem`**.

**Лут и инвентарь — граница ответственности:** **`LootSystem`** (или конвейер подбора) **только предлагает** подбор в мире: **`reserved`**, радиус / hit-test, таймаут, «кто подбирает», переходы **`LootState`**. **`InventoryService`** **единственный решает**, влезет ли предмет в сумку. Цепочка: мир готов к попытке → **`r = tryAddItem(...)`** → при **`r.ok`** — **`emitLootGranted`** (если нужен для потребителей **«Intent → Events»** / **«Events → State»** тика, телеметрии, реплея), **`picked` / `despawning`**; при **`!r.ok`** — лут обратно в **`idle`**, при желании UX по **`r.reason`** — см. [post-mvp-development-plan.md](./post-mvp-development-plan.md), фазы **2** и **5**.

**Следующий шаг:** WebSocket; сервер хранит комнату, принимает только эти события, валидирует, рассылает снапшоты. ECS на сервере без Pixi (те же компоненты-числа).

## Реализация: типовые риски и смягчение

| Риск | Как закрыть |
|------|-------------|
| **Дребезг FSM** (AI: walk → idle → walk каждый кадр) | **`minHoldTime`**; выход на слабее — только если **`time / duration >= interruptAt`** (per-анимация), плюс **`ANIMATION_COMPLETE`** для финала (**death** = **`interruptAt: 1`**). См. post-mvp 1.1. |
| **Смешение боевой логики и `Animation.state`** | Канон боя — **`CombatState` / `Health`**; клипы — производные. См. таблицу «Gameplay state и animation state» выше. |
| **Урон / HP из анимации** | Запрещено: анимация **только визуал**; см. блок «Анимация не влияет на gameplay». |
| **Неизвестный `Animation.state` / нет клипа** | **`defaultState = 'idle'`**; иначе залипание FSM. См. блок «Fallback» выше. |
| **Конфликт AI / ввод / knockback** | **`DesiredVelocity`** пишет **один** агрегирующий проход; **knockback** → **`ExternalForces` / импульсы**, не в **`DesiredVelocity`**. |
| **AI: все враги «думают» в одном тике** | **`thinkOffset`** (**`tickId % N`**) и/или **`AIGroupThink`** + **`nextThinkTime`** — post-mvp фаза 3. |
| **Повторное применение одного `DamageEvent`** (двойной вызов фазы 3, падение, реплей) | **Обязательно:** уникальный **`eventId`** + **`processedEvents`**; если **`eventId`** уже обработан → **игнор**. См. контракт выше. |
| **Двойной / пропущенный урон** (два слоя эмиттят или порядок систем плавает) | **`emitDamage`** только из **`DamageEventEmitter`** внутри **одного** конвейера; остальные — **`AttackIntent`**. См. выше. |
| **`CombatSystem` раздувается** (всё в одном месте) | Разнести **`TargetResolution` / `DamageCalculation` / `DamageEventEmitter`**; оркестратор — тонкий. См. блок «не монолит» выше. |
| **Лут `picked`, но инвентарь не принял** | Только **`tryAddItem` → ok`** перед **`picked`**; **`reserved`** до успеха — см. post-mvp фаза 2. |
| **Дублирование правил сумки в `LootSystem`** | Стак / лимиты / квесты — **только** в **`InventoryService`**; лут **предлагает**, сервис **решает** + **`reason`** при отказе. |
| **Лут `reserved`, резервирующий умер / удалён** | **Обязательно каждый тик:** **`!entityExists(reservedBy)`** → **`LootState = idle`**, **`reservedBy = null`**; иначе **залипание**. См. post-mvp фаза 2. |
| **Задержка `poll()` на 1 тик** для VFX | Опционально **`lane: 'cosmetic'`** на `RenderEvent` — обработка сразу; логика смерти/FSM — **gameplay**, N+1. |
| **«Где лежат события»** | Структура **`GameEventQueues`** / **`EventQueue<T>`** на контексте мира. |
| **Очереди: double-consume / порядок** | **Double-buffer** + **фасад** (`emit*` → `current`, `get*` → `processing`); без голых массивов. См. блок выше. |
| **`processedEvents` раздувается** | **`Map<tickId, Set<eventId>>`**, выкидывать тики старше **`currentTick - N`**. |
| **Спам / гонка `AnimationRequest`** | За тик на **`eid`**: **`bestByPriority`** (числа **100 / 80 / 60 / 20 / 0**), см. раздел анимации. |
| **HP-бары + lerp камеры + float позиция** | Предпочтительно **screen-space** слой: **`worldToScreen` → `Math.round` / snap к целому пикселю**; мир + сабпиксель без округления даёт **subpixel jitter**. |
| **RenderSync трогает ECS-сущности** | Запрещено: только реестр Pixi. |
| **Весь VFX, камера, твины HUD в одном `RenderSync`** | Вынести в отдельные модули **presentation**; иначе слой «мост» ломается при каждой фиче — см. «Три слоя» выше. |
| **`RenderAdapter` мутирует ECS** | Только **read ECS** + **эмит в очередь**; мутация — только системы после **`poll()`**. |
| **Двойной источник правды (UI / кэш)** | Канон: **ECS = мир**, **`playerState` = инвентарь (+мета)**; React и прочие кэши — **только derived**, см. блок выше. |

## Telegram Web Apps — обязательные учёты

- **Viewport:** `Telegram.WebApp.expand()` при старте; слушать `viewportChanged` для ресайза canvas.
- **Жесты:** не перехватывать вертикальный скролл всей страницы — canvas на весь экран, `touch-action: none` на контейнере игры.
- **Клавиатура:** стрелки — для десктопа / внешней клавиатуры; на телефоне основной ввод — **виртуальный джойстик** или **тап по клетке / в направлении** (для MVP достаточно: тап = идти в сторону тапа или короткий шаг по сетке).
- **Безопасность:** позже — проверка `initData` на сервере для привязки прогресса; в MVP не блокер.

```
game-rpg/
  apps/client/          # Vite + TypeScript + Pixi + bitECS
  apps/server/          # express/fastify, static + API
```

Монорепо не обязателен: можно начать с одного `client/` и `server/` в корне.

## Риски и смягчение

| Риск | Смягчение |
|------|-----------|
| Версии **pixi.js** и **pixi-tiledmap** | Ставить последние совместимые релизы одной командой (`npm i pixi.js pixi-tiledmap`); при регрессии — зафиксировать пары версий в lockfile |
| Пути в `.tmj` | Все URL от корня сайта с префиксом `/assets/`; зеркало на диске `public/assets/` |
| Двойной источник правды (клиент/сервер) | С первого дня проектировать `MOVE` / `ATTACK` / `USE_ITEM` и недоверие к «результату» с клиента; при появлении сервера — он авторитетен для позиций и урона |
| Двойной источник на клиенте (UI vs ECS) | **ECS / `playerState`** — канон; UI-кэши — **только derived**; см. «Единственный источник правды» выше |
| FSM дергается на мобиле | `minHoldTime`, см. раздел «Реализация: типовые риски» |
| Путаница скорости AI / knockback | Один агрегатор **`DesiredVelocity`** + **`ExternalForces`**, затем **`ActualVelocity`**; см. выше |
| Много врагов — всплеск CPU на **`think()`** | **`thinkOffset`** / **`tickId % N`** или группы (**`AIGroupThink`**); post-mvp фаза 3 |
| Перегруженный combat pipeline | Resolution / расчёт / **`DamageEventEmitter`** отдельно; оркестратор тонкий — см. «Единственный продьюсер» |
