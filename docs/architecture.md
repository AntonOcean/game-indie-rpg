# Архитектура MVP: 2D RPG для Telegram Web Apps

Цель: изометрический «вайб» Diablo 2 на ортогональной сетке с минимальным боем и лутом. Клиент — **mobile-first** (тапы + опционально стрелки на десктопе). Сервер — **Node.js** для будущей авторитативности; на этапе MVP допустим **клиент-авторитет** с валидацией там, где это дёшево.

## Стек и роли

| Слой | Технология | Роль |
|------|------------|------|
| Рендер | [PixiJS v8](https://pixijs.com/) (последний стабильный `pixi.js` на npm) | WebGL / WebGPU, `Application`, Assets, сцена, интерактив |
| Логика сущностей | [bitECS](https://github.com/NateTheGreatt/bitECS) | ECS: позиции, HP, теги, запросы в системах |
| Карта | [**pixi-tiledmap**](https://www.npmjs.com/package/pixi-tiledmap) | Загрузка `.tmj` / Tiled JSON под Pixi v8, слои → display tree |
| Ассеты MVP | `assets/` | Урезанный набор: одна карта, тайлсеты по ссылкам из `.tmj` |
| Полный пак | `full-assets/` | Источник для расширения; не тащить в бандл целиком |

## Связка ECS ↔ рендер (адаптер, не жёсткий 1:1)

**Правило:** слой ECS **не импортирует** типы Pixi и не держит ссылки на `DisplayObject`. Вместо «один `eid` = один спрайт навсегда» используется косвенная связь.

**Компонент (пример):** `RenderRef { renderId }` — непрозрачный идентификатор (число / UUID) записи в **реестре представлений** снаружи ECS (`Map<renderId, Container | Graphics>` или пул узлов). Один `eid` может иметь несколько `RenderRef` (тень + тело + оружие), либо несколько сущностей с общим визуальным корнем — на выбор команды; важно, что ECS знает только `renderId`.

**Разделение ответственности (чтобы не «протыкать» ECS из Pixi):**

1. **`RenderSyncSystem`** (имя может совпадать с привычным `RenderSystem`, но смысл узкий) — **только синхронизация ECS → реестр → Pixi**: читает `Position`, `Health`, флаги видимости, состояние анимации в компонентах и т.д., обновляет или пересоздаёт **узлы реестра** по `renderId`. **Не** создаёт и **не** удаляет **сущности ECS** (`addEntity` / `removeEntity` / компоненты жизненного цикла) — иначе появятся скрытые сайд-эффекты и рассинхрон мира и картинки. Движение и бой **не** трогают `sprite` напрямую.

2. **`RenderAdapter` (мост Pixi → игра)** — **единственная точка**, где события от **рендера** ставятся в очередь; игровой слой их **не подписывает напрямую** на `sprite.on(...)`.

### Контракт RenderAdapter (формальный)

Типы событий (имена и поля можно слегка расширить, но принцип — **discriminated union** + один вход в ECS):

```ts
// entity — bitECS eid (number)
// lane: опционально; по умолчанию вести как gameplay (обработка в тике N+1).
type RenderEvent =
  | { type: 'ANIMATION_COMPLETE'; entity: number; lane?: 'gameplay' | 'cosmetic' }
  | { type: 'POINTER_TAP'; worldX: number; worldY: number; lane?: 'gameplay' | 'cosmetic' }
  | { type: 'SPRITE_READY'; entity: number; lane?: 'gameplay' | 'cosmetic' }

interface RenderAdapter {
  /** Снять все события, накопленные с прошлого кадра (или с момента последнего poll). */
  poll(): RenderEvent[]
}
```

**Правило:** обработчики Pixi (`complete`, `pointertap`, `texture` loaded, …) **только** пишут во **внутреннюю очередь** адаптера. **ECS-системы и остальной игровой код** читают рендер **только** через `poll()` (момент вызова — по **фазам тика** ниже: конец тика и/или начало следующего — зафиксировать в коде). Запрещено «ой, тут из `sprite.on()` сразу дерну `addComponent` / мир» вне этого контура — иначе контракт ломается.

**Лут и VFX:** **`RenderAdapter` не должен определять судьбу сущности** (удаление лута, выдача в инвентарь). Подбор → состояние + **игровой таймер** на сущности → **RenderSync** рисует fade; по обнулению таймера **игровая система** удаляет entity. См. [post-mvp-development-plan.md](./post-mvp-development-plan.md), фаза 2 (Loot VFX).

Соответствие примерам:

| Событие Pixi | `RenderEvent` |
|--------------|----------------|
| конец клипа анимации | `ANIMATION_COMPLETE` |
| тап по `DisplayObject` (мир уже переведён в `worldX` / `worldY`) | `POINTER_TAP` → дальше тот же слой, что **`PlayerIntent`** / hit-test |
| текстура/спрайт готов, известны размеры кадра | `SPRITE_READY` → система обновляет **`Hitbox`** и т.д. |

**Импорт и мутация `DisplayObject` для игровых сущностей** остаются в коде рендера (sync + адаптер + фабрика узлов реестра), но **семантически**: sync только «рисует состояние мира», adapter только «копит и отдаёт `RenderEvent[]`».

**Задержка в 1 тик (N → N+1):** для событий, влияющих на **FSM / логику смерти**, политика **gameplay** — consume в **N+1**. Для **чисто косметики** (ударный блеск, не связанный с `DamageEvent`) можно пометить **`lane: 'cosmetic'`** и обрабатывать **сразу** после `poll()` в конце кадра N, чтобы уменьшить ощущемую задержку — не смешивать с удалением entity и не нарушать единый поток без явной метки.

**Зачем:** несколько визуальных элементов на сущность, пересоздание спрайта без пересоздания `eid`, скрытие / замена вида без протекания API рендера в боевую логику — и без разрастания вызовов `addComponent` из обработчиков по всей сцене.

## Ввод: Input → Intent

Сырые события (тап, стрелки, pointer) **не** напрямую двигают игрока и **не** сразу наносят урон. Отдельный слой собирает кадр (или тик) в структуру **намерения**:

```ts
// Пример формы; поля опциональны по кадру
type PlayerIntent = {
  moveTo: { x: number; y: number } | null       // десктоп: куда идти (цель в мире)
  moveDirection: { x: number; y: number } | null // мобила: нормализованное направление от тапа
  attackTarget: number | null                    // eid цели или null (bitECS entity id)
}
```

Сборщик ввода выставляет **либо** целевую точку, **либо** направление (см. таблицу desktop/mobile и правило **остановки** движения на мобиле в [implementation-plan.md](./implementation-plan.md)). **Системы решения** (порядок фиксирован): проверка валидной атаки по `attackTarget`; иначе движение по `moveDirection` или к `moveTo`. Так проще добавлять `USE_ITEM`, приоритеты, станы, GUI без переписывания низкоуровневого ввода.

## Намерения анимации (рост после MVP)

Бой, AI и ввод **не** меняют напрямую `Animation.state` у сущности — иначе FSM разъедут несколько авторов. Вместо этого в очередь попадает **запрос**:

```ts
type AnimationRequest = {
  entity: number
  state: 'idle' | 'walk' | 'attack' | 'hurt' | 'death'
  force?: boolean // редко: пробить часть блокировок (использовать осознанно)
}
```

**`AnimationSystem` (FSM)** читает запросы и текущий `Animation`, решает переход по **приоритетам** (`death > hurt > attack > walk > idle`). События из **`RenderAdapter`** (`ANIMATION_COMPLETE`) **не** пишут в `Animation` напрямую — переводятся в запросы/флаги в **одной** точке. Детали: **`minHoldTime`**, **`interruptAt`** (окно прерывания клипа, 0..1), анти-дребезг — [post-mvp-development-plan.md](./post-mvp-development-plan.md), фаза 1.1.

**Несколько `AnimationRequest` за один тик на одну сущность:** ❌ не складывать в FIFO-очередь без правила — порядок вызовов систем даст **недетерминизм** (например **attack** от игрока и **hurt** от врага в одном кадре). ✅ За тик на **`eid`** держать **ровно один** агрегированный запрос: **`AnimationIntentBuffer[eid] = bestByPriority(requests)`**, числовые приоритеты состояний (выше = жёстче): **`death: 100`**, **`hurt: 80`**, **`attack: 60`**, **`walk: 20`**, **`idle: 0`**. При равном приоритете — **детерминированное** правило (например порядок источников в коде или **`seq`** на запросе). Уже **после** этого сравнивать с текущим **`Animation.state`**, **`locked`**, **`minHoldTime`**, **`interruptAt`**.

**Анти-spam (дополнительно):** если **`requestedState === currentAnimation.state`** без смысла перезапуска клипа — не обновлять буфер дубликатом. Иначе AI засыпает **`walk, walk, …`**.

**Направление (`Facing`) — не часть `Animation.state`:** держать отдельным компонентом (угол или `up | down | left | right`). Иначе FSM раздуется до `walk_up`, `walk_down`, … Вместо этого: **`state: walk`** + **`facing: left`** (зеркало спрайта в RenderSync). **`facingLocked` — когда снять:** явное правило, иначе персонаж «залипнет» разворотом после атаки. Варианты (выбрать один в коде и не смешивать): (1) **`facingLocked = (state === 'attack' && time / duration < interruptAt)`** и аналог для **`hurt`** по их **`interruptAt` / `minHoldTime`**; **`idle` / `walk`** — всегда разблокировано; **`death`** — обычно **`Animation.locked`** до конца клипа. (2) **`facingLocked := Animation.locked`** для клипов, где поворот запрещён целиком. Пока **`facingLocked`**, ввод **не** меняет **`Facing`**.

## Движение: желаемая и фактическая скорость

Чтобы **AI**, **ввод**, **knockback**, **slow** и **stun** не ломали друг друга, развести:

| Компонент / поле | Кто пишет | Смысл |
|------------------|-----------|--------|
| **`DesiredVelocity`** | **Один** проход / **одна** система намерения движения (например **`MovementIntentSystem`**) | Итог «куда хочет идти сущность» после агрегации **ввода**, **`AIIntent`**, станнов (обнуление). ❌ **Нельзя**, чтобы **AI**, **ввод** и **knockback** **независимо** писали в одно поле — последний перетрёт остальных. |
| **`ExternalForces` / `ImpulseBuffer`** (имя условное) | Бой, **knockback**, отталкивания | Вклады, которые **не** являются «намерением идти»; суммируются в **`MovementSystem`** при расчёте фактической скорости. |
| **`ActualVelocity`** (факт за кадр) | **Movement + Collision** (один слой) | После **`DesiredVelocity` + внешних сил**, коллизий, модификаторов. |

Интеграция позиции: **`Position += ActualVelocity * dt`** (или эквивалент с разрешением коллизий по шагам). **Инвариант:** **`Position` напрямую меняет только `MovementSystem`** (или один согласованный модуль «физика мира»), который учитывает **`ActualVelocity`**, коллизии, телепорты уровня дизайна. ❌ **Knockback не через перезапись `DesiredVelocity` у AI** — иначе AI и отбрасывание будут бороться за одно поле. Подробнее — [post-mvp-development-plan.md](./post-mvp-development-plan.md), фаза 3.

**Урон от AI:** система AI **никогда** не пушит **`DamageEvent`** напрямую. Цепочка: **`AIIntent`** (в т.ч. `type: 'attack'`) → общий контур **`AttackIntent` / валидация** (как у игрока) → **`DamageEvent`**. Так не дублируется логика дистанции/кулдауна и нет двойной генерации урона.

## Клиент: слои приложения

1. **Bootstrap TWA** — `Telegram.WebApp.ready()`, размер viewport, отключение зума страницы при необходимости, CSS `100dvh` / safe-area.
2. **Asset pipeline** — расширения **pixi-tiledmap** для `Assets` в Pixi v8; файлы под **`public/assets/`**, в `.tmj` и тайлсетах — пути **с префиксом `/assets/`** и `fetch('/assets/map.tmj')` без кастомного basePath (см. [implementation-plan.md](./implementation-plan.md), фаза 2.1).
3. **Карта** — через **pixi-tiledmap**: визуальные слои `ground`, `wall`; слой `collisions` скрыть (в Tiled или `visible = false` на контейнере слоя). Массив `data` слоя `collisions` читать для **физики** (или экспортировать отдельный буфер при загрузке).
4. **ECS-мир** — отдельный объект `gameWorld` (не путать с `createWorld` из bitECS — можно назвать `ecsWorld`). Компоненты в SoA: **`Position { x, y }` в мировых пикселях** (как Tiled), `Velocity`, `Health`, `Hitbox`, флаги `Player`, `Enemy`, `Loot`, `Dead` и т.д. Тайлы — только через `floor` при проверке сетки (см. [implementation-plan.md](./implementation-plan.md)). Рекомендуемые **теги** (или отдельные tag-компоненты): **`TagPlayer`**, **`TagEnemy`**, **`TagLoot`** — дешёвые фильтры в системах и отладка.
5. **Игровой цикл (тик)** — один **зафиксированный порядок фаз**; время из тикера (`deltaMS` → секунды, кап — см. [implementation-plan.md](./implementation-plan.md)).

### Фазы тика (tick order)

| # | Фаза | Смысл |
|---|------|--------|
| 1 | **Input → Intent** | Сырой ввод → **`PlayerIntent`** (`moveTo`, `moveDirection`, `attackTarget`). |
| 2 | **Intent → Events** | Валидация намерений → очередь **игровых** событий (`DamageEvent`, `LootGranted`, …); не смешивать с немедленным списанием HP в том же проходе. |
| 3 | **Events → State changes** | Применение событий: **`Health`**, инвентарь, смерть → лут, таймеры VFX лута, кулдауны. |
| 4 | **Movement + Collision** | Смещение позиций, коллизии с сеткой (игрок и прочие, кому положено в этой фазе). |
| 5 | **AI** | Логика врагов **после** обновления намерений/событий/движения за кадр (преследование, атака, stuck — см. post-mvp). |
| 6 | **RenderSync** | Синхронизация реестра по **`RenderRef`**, **`Position`**, визуальным компонентам; только отображение состояния мира. |
| 7 | **`RenderAdapter.poll()`** | Снять **`RenderEvent[]`** из очереди адаптера. |

**Зачем так:** предсказуемость, меньше скрытых гонок, проще вставлять системы — всегда видно, до или после движения / AI они выполняются.

**`GameEventQueues` и этот порядок:** в кадре **N** фаза **2** пишет только в **`current`**; фаза **3** читает **`processing`** (содержимое с кадра **N−1**, после **`swap()` в конце N−1**), затем **`processing.clear()`**. В **конце кадра N** (логично сразу после фазы **7** и обновления **`tickId`**) — **`swap()`**, чтобы накопленное в **`current`** стало **`processing`** для кадра **N+1**. Иначе фазы **2** и **3** в одном кадре конфликтуют с одним массивом. Детали — блок «Где живут события» ниже.

**Политика шага 7 (`poll()`) — зафиксировать явно:**

```text
Кадр N: фазы 1–6 → RenderAdapter.poll() → события попадают в буфер игры (очередь на следующий тик)

Кадр N+1: в начале тика (до или вместе с фазой 1–2) — consume / раздача накопленных RenderEvent
```

Так меньше **недетерминизма** и реже ситуация «`ANIMATION_COMPLETE` то срабатывает, то нет» из-за порядка колбэков внутри кадра. Если нужна обработка **в том же** кадре N — документируйте **другой** контракт (например второй `poll()` в начале N+1 не используется, а события сливаются сразу после фазы 6) и придерживайтесь его везде.

**Камера** — после фазы 6: след за игроком, clamp к карте; опционально **`cameraLerp`** и **`cameraDeadzone`** (например `{ x: 40, y: 30 }` в px экрана), чтобы не было дрожания при микродвижениях — [post-mvp-development-plan.md](./post-mvp-development-plan.md).

**MVP (сопоставление):** смерть врага и спавн лута — в фазе **3** после применения урона; подбор лута — намерение во **2** (или 1), результат в **2–3**.

### Время (GameTime): один источник истины

Все системы берут время из **одного** контекста тика (имя условное):

```ts
type GameTime = {
  dt: number      // длительность тика в секундах (из Ticker / performance, с верхним капом)
  now: number     // монотонные часы в секундах (или ms — но тогда везде ms)
  tickId: number  // инкремент +1 каждый игровой тик (логи, события, см. DamageEvent)
}
```

**Правило:** не вводить параллельно «таймеры в миллисекундах» и «в секундах» в разных модулях. Примеры согласования с **`dt`** (в секундах):

- `animation.time += gameTime.dt`
- `pickupVfxTimer -= gameTime.dt`
- `attackCooldown -= gameTime.dt`

Иначе через несколько фаз появятся рассинхрон анимации и логики и ошибки масштаба времени.

## Коллизии (MVP)

Текущая карта (`assets/map.tmj`): ортогональная сетка **32×32**, слой **`collisions`** — тайлы с ненулевым gid трактуются как **непроходимые** (например gid `30` по всему периметру и препятствиям).

Алгоритм:

- **Игрок / враг / лут** — ось-выровненный **AABB** в **мировых пикселях** (квадрат в мире = один размер, например 24×24 с отступом от клетки).
- **Доступ к сетке:** `isBlockedTile(tx, ty)`; после смещения — проверка **четырёх углов** хитбокса → тайлы через `floor(world / tileSize)` (детали в плане имплементации).
- **Ввод:** один перевод **screen → world** с учётом камеры и `worldScale`, чтобы клики и `Position` были в одной системе.
- **Разрешение:** часто пробное смещение по X, затем по Y с откатом оси; в углах возможны застревание и дрожание — при необходимости сеточный шаг или мелкие подшаги (см. план имплементации).
- Альтернатива позже: object layer с полигонами — не нужна для MVP с квадратами.

## Бой и лут (MVP)

- **Атака:** тап → мировые координаты → в **`PlayerIntent`** выставить `attackTarget` или поля движения. Урон только если одновременно: цель **живая** (`Enemy`, не `Dead`), **дистанция \< attackRange**, **кулдаун** (порядка 300 ms) истёк — иначе без урона. **Не привязывать урон к кадру анимации** (см. ниже и [post-mvp-development-plan.md](./post-mvp-development-plan.md)).
- **Смерть:** `Health <= 0` → компонент `Dead`, отключение коллизии врага, удаление/скрытие спрайта → создать сущность лута в той же позиции.
- **Лут:** один тип (например «монета») — квадрат другого цвета; подбор при пересечении с игроком (или тап по луту — на мобиле удобнее авто-подбор при касании).

## Сервер Node.js (MVP и рост)

**MVP:** сервер раздаёт статику (Vite build / `dist`) и опционально JSON health. Игровое состояние полностью на клиенте (**клиент-авторитет** — допустимо для одиночного MVP).

**Заложить сразу (античит-мышление, даже без логики на сервере):**

- не принимать с клиента «голый» урон или смерть цели как истину; в проде урон считает сервер по своему `ATTACK` и таблицам;
- не принимать произвольные телепорты позиции; в проде `MOVE` валидируется по скорости, dt, коллизиям с миром сервера;
- клиент может **отображать** предикт, но канон — версия сервера (когда появится).

**Единый контракт событий игрока** — завести типы/enum и очередь на клиенте уже в MVP (хотя бы `console` или no-op отправитель), чтобы потом тот же payload уйти по WebSocket:

| Событие | Назначение (смысл для будущего сервера) |
|---------|----------------------------------------|
| `MOVE` | намерение сместиться (цель, направление или снапшот ввода — уточнить схему; сервер проверяет допустимость) |
| `ATTACK` | намерение ударить цель `targetId` (сервер проверяет дистанцию, кулдаун, LOS) |
| `USE_ITEM` | использование предмета слота (сервер проверяет наличие, кулдаун, контекст) |

Тела сообщений — отдельные TypeScript-типы в общем пакете `shared/` или `packages/protocol/`, чтобы клиент и Node использовали одни и те же имена полей.

**Внутренний пайплайн боя и лута на клиенте (готовность к авторитетному серверу):** по возможности **не** применять урон и смерть в той же системе, которая только «решила атаку». Развести этапы: намерение / валидация (**`ATTACK`**, **`PICKUP`**) → **событие результата** → система, которая **меняет `Health` / инвентарь / мир**.

**Где живут события:** явный слой очередей с **жёстким double-buffer** (must-have для фазы 5 и сервера — одинаковый порядок, нет случайного double-consume, удобный лог **`processing`**):

```ts
type GameEventQueues = {
  /** Писать только сюда в фазе 2 текущего кадра. */
  current: { damage: DamageEvent[]; loot: LootGranted[] /* … */ }
  /** Читать только отсюда в фазе 3 следующего кадра (после swap). */
  processing: { damage: DamageEvent[]; loot: LootGranted[] /* … */ }
}
// swap(): поменять current ↔ processing в конце кадра (после фазы 7 / перед инкрементом tickId)
```

**API без прямого доступа к массивам:** ❌ не экспортировать **`queues.current.damage.push`** наружу — фаза **3** или ошибка продьюсера могут **писать в `processing`** и сломать порядок. ✅ Обернуть в объект с методами вроде **`emitDamage(e)`** / **`emitLootGranted(…)`** (всегда пишут **только** в **`current`**) и **`getDamageEvents()`** / **`getLootGranted()`** для потребителей фазы **3** (возвращают **только** срез **`processing`**, read-only или копия до **`clear`**). Внутренняя структура — **приватная**.

**Контракт по кадрам:**

```text
Кадр N:
  фаза 2 — продьюсеры пушат только в queues.current

Конец кадра N:
  queues.swap()   // то, что накопили в current, становится буфером для чтения в N+1

Кадр N+1:
  фаза 2 — новые события пишутся в queues.current (уже «чистый» после swap)
  фаза 3 — потребители читают только queues.processing, применяют изменения мира
  после фазы 3 — queues.processing.clear() на каждую внутреннюю очередь (damage, loot, …)
```

Не смешивать «писать и читать один массив в одном кадре» без такого разделения — иначе легко **дважды применить** событие или потерять порядок относительно будущего серверного тика.

Пример **`DamageEvent`**:

```ts
type DamageEvent = {
  tickId: number
  eventId: string // уникально в рамках сессии: `${tickId}-${seq}` или uuid
  sourceType: 'entity' | 'environment' | 'dot'
  sourceId: number // при sourceType === 'entity' — eid; иначе зарезервированный id зоны/ловушки/тика DoT (явная схема, без «-1»)
  targetId: number // eid цели
  amount: number
  sourceX: number
  sourceY: number
}
```

- **`tickId` + `eventId`:** в одном тике может быть **несколько** попаданий; одного `tickId` недостаточно. **`HealthSystem`** ведёт **скользящее окно** по тикам, чтобы **`Set` не рос бесконечно**, например **`Map<tickId, Set<eventId>>`**: при продвижении **`currentTick`** удалять записи с **`tickId < currentTick - N`** (**N ≈ 30–60** тиков достаточно для защиты от повторной доставки). Проверка: **`eventId`** уже есть в множестве для своего **`tickId`** → пропуск. Альтернатива — один глобальный **`Set<eventId>`** с LRU по размеру, если **`eventId`** глобально уникален (uuid).
- **`sourceType`** — ловушка, зона, DoT и сущность различаются явно; **`sourceId`** + **`sourceX/Y`** согласованы с типом (для **`environment`** — якорь зоны в мире).
- **`sourceId` / `targetId`** при **`entity`** — агро, «кто убил», несколько источников за тик; логи и реплей.
- **`sourceX` / `sourceY`** — позиция источника в мировых пикселях (отбрасывание, VFX направления удара, попапы урона); для статических ловушек можно брать центр зоны или последнюю известную точку источника.

**Урон и анимация:** цепочка **`AttackIntent` → валидация → `DamageEvent` → `Health`**. Клип атаки — **только визуал**. ❌ Не наносить урон с **конкретного кадра спрайта** или `onFrame` — иначе авторитетный сервер и сетевая сверка сломают модель.

Тогда при появлении сервера каноничными становятся **события результата** с сервера (или подтверждённые дельты), а не перепись всего боя. Подробнее — [post-mvp-development-plan.md](./post-mvp-development-plan.md) (фаза 5 и принципы).

**Граница `playerState` и ECS (источник истины):**

| Данные | Где живут |
|--------|-----------|
| `Position` игрока (и прочих существ в мире) | **ECS** |
| `Health` игрока (и врагов) | **ECS** |
| Инвентарь / сумка | **`playerState.inventory`** (не ECS) |
| UI (HUD, оверлеи) | Читает **ECS + `playerState`**; не дублировать HP/позицию только в React-состоянии без синхронизации с ECS |

**Правило:** **`playerState` не хранит данные мира** — не копировать в него позицию, HP, состояние врагов. Иначе неизбежен рассинхрон; в сейв уходит снимок по правилам: мир из ECS, мета-прогресс из `playerState`.

**Инвентарь после MVP:** сущности **`Loot` в мире** остаются в ECS; **содержимое сумки** — в **`playerState`** (например `inventory: ItemStack[]`), не набор сущностей-слотов в ECS.

**API инвентаря:** не мутировать массив инвентаря из множества систем напрямую. Один **транзакционный** вход: **`addItem(...) : boolean`** — внутри **проверка + добавление** в одном методе (без дублирования «сначала canAdd, потом add»). При `false` инвентарь не меняется. Плюс **`removeItem`**, **`hasItem`**. Подбор лута: только после **`addItem === true`** менять состояние лута — см. [post-mvp-development-plan.md](./post-mvp-development-plan.md), фаза 2. (Имя **`tryAdd`** допустимо как синоним, если это тот же атомарный метод.)

**Следующий шаг:** WebSocket; сервер хранит комнату, принимает только эти события, валидирует, рассылает снапшоты. ECS на сервере без Pixi (те же компоненты-числа).

## Реализация: типовые риски и смягчение

| Риск | Как закрыть |
|------|-------------|
| **Дребезг FSM** (AI: walk → idle → walk каждый кадр) | **`minHoldTime`**; выход на слабее — только если **`time / duration >= interruptAt`** (per-анимация), плюс **`ANIMATION_COMPLETE`** для финала (**death** = **`interruptAt: 1`**). См. post-mvp 1.1. |
| **Конфликт AI / ввод / knockback** | **`DesiredVelocity`** пишет **один** агрегирующий проход; **knockback** → **`ExternalForces` / импульсы**, не в **`DesiredVelocity`**. |
| **Несколько ударов в одном `tickId`** | Поле **`eventId`** + **`processedEvents`** в системе применения урона. |
| **Лут `picked`, но инвентарь не принял** | Атомарный **`addItem` → boolean`**; **`reserved`** до успеха — см. post-mvp фаза 2. |
| **Лут `reserved`, резервирующий умер / удалён** | Если **`!entityExists(reservedBy)`** → **`idle`**, **`reservedBy = null`**; иначе лут **залипает** навсегда. |
| **Задержка `poll()` на 1 тик** для VFX | Опционально **`lane: 'cosmetic'`** на `RenderEvent` — обработка сразу; логика смерти/FSM — **gameplay**, N+1. |
| **«Где лежат события»** | Структура **`GameEventQueues`** / **`EventQueue<T>`** на контексте мира. |
| **Очереди: double-consume / порядок** | **Double-buffer** + **фасад** (`emit*` → `current`, `get*` → `processing`); без голых массивов. См. блок выше. |
| **`processedEvents` раздувается** | **`Map<tickId, Set<eventId>>`**, выкидывать тики старше **`currentTick - N`**. |
| **Спам / гонка `AnimationRequest`** | За тик на **`eid`**: **`bestByPriority`** (числа **100 / 80 / 60 / 20 / 0**), см. раздел анимации. |
| **HP-бары + lerp камеры + float позиция** | Предпочтительно **screen-space** слой: **`worldToScreen` → `Math.round` / snap к целому пикселю**; мир + сабпиксель без округления даёт **subpixel jitter**. |
| **RenderSync трогает ECS-сущности** | Запрещено: только реестр Pixi. |

## Telegram Web Apps — обязательные учёты

- **Viewport:** `Telegram.WebApp.expand()` при старте; слушать `viewportChanged` для ресайза canvas.
- **Жесты:** не перехватывать вертикальный скролл всей страницы — canvas на весь экран, `touch-action: none` на контейнере игры.
- **Клавиатура:** стрелки — для десктопа / внешней клавиатуры; на телефоне основной ввод — **виртуальный джойстик** или **тап по клетке / в направлении** (для MVP достаточно: тап = идти в сторону тапа или короткий шаг по сетке).
- **Безопасность:** позже — проверка `initData` на сервере для привязки прогресса; в MVP не блокер.

| Риск | Смягчение |
