# План имплементации MVP

Скоуп: квадратный игрок, квадратный враг, одна карта (`assets/map.tmj`), один тип удара, один тип лута. Управление: **стрелки** (десктоп) + **тапы** (мобила). Коллизии по слою **`collisions`**.

## Фаза 0: Репозиторий и сборка

1. Инициализировать **client** (рекомендуется Vite + TypeScript).
2. Зависимости:
   - **`pixi.js@8`** — последний стабильный минор/патч восьмой ветки (`npm install pixi.js@8`),
   - **`pixi-tiledmap`** — загрузка Tiled под Pixi v8 (peer к `pixi.js`),
   - `bitecs`.
   Fallback при крайней необходимости: `fetch('map.tmj')` + ручная отрисовка тайлов (логика коллизий не меняется).
3. **Ассеты только через `public/`:** зеркало в **`public/assets/`** (файлы на диске: `public/assets/map.tmj`, `public/assets/tilesets/...` и т.д.). См. правило путей в карте — фаза 2.1.
4. **Server:** минимальный `express` (или `fastify`) на порту 3000: `app.use(express.static('dist'))` после сборки клиента; в dev режиме клиент отдельно (`vite dev`), сервер только для прокси/API по желанию.
5. **Протокол (заготовка):** модуль `shared/` или `src/protocol/` с типами событий `MOVE`, `ATTACK`, `USE_ITEM` и функцией-заглушкой «отправить на сервер» (пока no-op) — см. [architecture.md](./architecture.md).

**Критерий готовности:** пустая сцена Pixi на весь экран, без ошибок в консоли, TWA script подключён (можно заглушкой).

## Фаза 1: Telegram Web App shell

1. Подключить `https://telegram.org/js/telegram-web-app.js`.
2. Вызвать `Telegram.WebApp.ready()` и `expand()`.
3. CSS: контейнер игры `width: 100%; height: 100%;` / `100dvh`, при необходимости `env(safe-area-inset-*)` для кнопок UI.
4. Canvas Pixi: `resizeTo: window` или ручной resize по `visualViewport` / `Telegram.WebApp.viewportStableHeight`.

**Критерий:** игра заполняет область мини-приложения на телефоне без «прыгающей» высоты.

## Фаза 2: Загрузка карты и отрисовка

### 2.1 pixi-tiledmap + пути Tiled (узкое место)

**Правило (жёстко):** все пути к ресурсам внутри экспортированного `.tmj` (и во внешних `.tsx`, если там есть ссылки на текстуры) должны быть **абсолютными от корня сайта** и начинаться с **`/assets/`**.

Примеры строк в JSON:

- `/assets/map.tmj` — точка входа в коде;
- `/assets/tilesets/TX Tileset Grass.tsx`;
- `/assets/tiles/grass.png`.

Тогда:

- загрузка карты: **`fetch('/assets/map.tmj')`** (или `Assets.load` с тем же URL);
- **не нужны** `basePath`, кастомный `pathResolver` и угадывание «относительно чего» резолвит лоадер — браузер сам запросит каждый вложенный URL как есть;
- на диске структура совпадает с URL: `public/assets/...` → после сборки `dist/assets/...`.

Перед коммитом: в Network все запросы к карте и цепочке тайлсетов — **200**. Если Tiled вставил пути без ведущего слеша — поправить в Tiled или скриптом постобработки экспорта.

### 2.2 Слои и константы

1. Подключить парсеры/лоадеры **pixi-tiledmap** к `Assets` приложения Pixi v8 (см. README пакета).
2. Загрузить карту с учётом п. 2.1.
3. Собрать карту в контейнер: слои **`ground`**, **`wall`** по порядку; слой **`collisions`** не показывать (скрыть или не монтировать в сцену).
4. Сохранить ссылку на массив **`collisions.data`**, `map.width`, `map.height` (в тайлах), `tilewidth`, `tileheight` — для коллизий и камеры.

**Критерий:** карта из репозитория видна целиком или с камерой (фаза 4).

## Фаза 3: ECS + квадратный игрок

### Единицы координат (зафиксировать до коллизий и камеры)

- **`Position.x` / `Position.y`** — **пиксели мира** в той же системе, что и слои Tiled (левый верх карты = `(0, 0)`, ось Y вниз как в большинстве 2D-редакторов). Никаких «тайловых» чисел в компоненте позиции.
- Переход **мир → индекс тайла** (для `isBlockedTile` и отладки):

  ```text
  tileX = floor(worldX / tileWidth)
  tileY = floor(worldY / tileHeight)
  ```

  `tileWidth` / `tileHeight` — из карты (у вас 32).
- **`Hitbox`** (смещение и размеры) в тех же **мировых пикселях**, что и `Position` (например центр сущности = `Position`, половины width/height — или явный `offset` в договорённой модели).

Так не смешиваются пиксели и тайлы: движение и камера считают в px, сетка коллизий — через `floor` и `isBlockedTile`.

1. `createWorld` (bitECS), зарегистрировать компоненты:
   - `Position`: `x`, `y` — **мировые пиксели** (см. выше),
   - `Player` (tag),
   - `RenderRef`: `renderId` (связь с реестром Pixi, без импорта Pixi в файлах компонентов),
   - `Hitbox`: `width`, `height` (или константа в коде для MVP),
   - опционально `Velocity` для плавного движения или шаг по сетке.
2. **Реестр представлений** + **`RenderSystem`:** создать для игрока узел (например `Graphics`, синий квадрат), положить в stage, записать `renderId` в `RenderRef`; каждый кадр синхронизировать позицию из ECS в реестр.

   **Правило:** **`RenderSystem` (и код создания/уничтожения узлов в реестре) — единственное место клиента, где импортируется и трогается Pixi** (`Container`, `Graphics`, `Sprite`, …). Системы движения, боя, лута работают только с компонентами ECS; если начать двигать `sprite` из боя или менять альфу сущности в `collisionResolve`, слой разъедется и отладка станет болью.
3. **Коллизии с тайлами — одна точка входа:** массив `data` слоя `collisions` — линейный; индекс **`ty * mapWidth + tx`** (тайловые координаты). Сразу завести утилиту и **не читать `data` наугад** из игровых систем:

   ```ts
   function isBlockedTile(tx: number, ty: number): boolean {
     if (tx < 0 || ty < 0 || tx >= mapWidth || ty >= mapHeight) return true
     return collisionData[ty * mapWidth + tx] !== 0
   }
   ```

   **Коллизия AABB ↔ тайлы — не по одному центру:** проверка только центра даёт проскоки в углах и «залипание». Закрепить правило: **четыре угла хитбокса** в мировых px → для каждого угла `tileX` / `tileY` через `floor` → четыре вызова `isBlockedTile`.

   ```ts
   function hitboxIntersectsBlockedTiles(hitbox: AabbWorldPx): boolean {
     const { left, top, right, bottom } = hitbox // в мировых пикселях
     const corners = [
       { x: left, y: top },
       { x: right, y: top },
       { x: left, y: bottom },
       { x: right, y: bottom },
     ]
     return corners.some((p) =>
       isBlockedTile(Math.floor(p.x / tileWidth), Math.floor(p.y / tileHeight))
     )
   }
   ```

   После пробного смещения сущности вызывать **`hitboxIntersectsBlockedTiles`** (или эквивалент `checkCorners`) и откатывать ось, если **любой** из углов попал в заблокированный тайл. **Вся** логика «стоит / не стоит» после движения — через эту цепочку, а не через один центральный тайл.

4. **Сборщик ввода → `PlayerIntent`:** разделить семантику **десктоп / мобила**, иначе на телефоне будет ощущение «тыкать куда бежать навсегда».

   | Устройство | Поведение |
   |------------|-----------|
   | **Desktop** | Клик / стрелки → в основном **движение к точке** в мире (`moveTo: { x, y }`, логика «идти к цели») |
   | **Mobile** | **Направление** от игрока к точке тапа — но обязательно определить **когда движение прекращается**, иначе персонаж «поедет бесконечно». |

   **Варианты для мобилы (выбрать один и описать в коде):**

   - **Удержание:** движение пока палец на экране (`pointerdown` → задаём `moveDirection`, `pointerup` / `pointercancel` → обнуляем);
   - **Короткий шаг:** один тап = импульс / фиксированное число тайлов или доли секунды, затем остановка;
   - **Стоп по отпусканию** в связке с виртуальным джойстиком (фаза 7).

   Без явного «стопа» direction-based модель непригодна.

   В типе намерения можно явно развести поля, например `moveTarget: { x, y } | null` и `moveDirection: { x, y } | null` (взаимоисключающие на кадр) или флаг `inputMode: 'point' | 'direction'`, выставляемый по `pointerType` / ширине / `Telegram.WebApp.platform`.

5. **Система разрешения намерения + движение:** по заполненным полям `PlayerIntent` выставить скорость или целевую точку; затем смещение `Position` **умножать на согласованный шаг времени** (см. блок **«Время (dt)»** ниже).

6. **collisionResolve (AABB vs тайлы):** классика — пробное смещение по **X**, затем по **Y**; после каждой оси проверять **четыре угла** хитбокса (`hitboxIntersectsBlockedTiles` / `checkCorners`), при `true` — откат этой оси. **Осторожно:** даже с углами в редких геометриях возможны **застревание** / **дрожание**; тогда — сеточный шаг, мелкие подшаги за кадр или сглаживание под ваш feel.

### Время (dt) в системах

В плане фигурирует «каждый кадр» — нужно явно задать **масштаб времени**, иначе на слабом телефоне всё будет тянуться, на десктопе 144 Hz — ускоряться.

**Минимум для MVP:**

- брать **реальный** `deltaTime` из тика Pixi (`Ticker.deltaMS` или разница `performance.now()`), в секундах: `dtSec = deltaMS / 1000`;
- скорости задавать в **единицах/сек** и применять `position += velocity * dtSec`;
- опционально **нормализация к «эталону 60 FPS»:** `const dtScale = deltaMS / 16.67` и умножать на неё величины, задуманные «на кадр при 60» (тогда при 30 FPS движение остаётся сопоставимым по ощущениям).

Продвинутый вариант позже: фиксированный шаг симуляции + накопитель времени; для MVP достаточно переменного `dt` с **верхней капой** (например не больше 50–100 ms за кадр), чтобы не проскакивать сквозь коллизии при лаге.

**Критерий:** квадрат ходит по проходимым клеткам, упирается в стены; коллизии через **`isBlockedTile`** и проверку **четырёх углов** хитбокса.

### Screen → world (pointer, клики по врагу)

Координаты из **Pixi `pointer*`** / `event.global` — в пространстве **стейджа** (или экрана относительно canvas). Чтобы получить **мировые пиксели** в той же системе, что **`Position`**, использовать **одну** функцию (например `screenToWorld(screenX, screenY)`), и для **ввода**, и для **атаки**, и для UI — иначе камера и scale разъедутся.

Если камера реализована смещением контейнера мира **`worldRoot`** и масштабом **`worldScale`** (`worldRoot.scale.x`):

```text
worldX = (screenX - worldRoot.x) / worldScale
worldY = (screenY - worldRoot.y) / worldScale
```

Здесь `screenX` / `screenY` — координаты той же системы, в которой заданы `worldRoot.x` / `worldRoot.y` (обычно глобальные координаты относительно `stage` / `Application.stage`). Если у вас вынесен отдельный «камерный» вектор, подставьте его вместо `worldRoot.position` — **главное**: формула везде одна: **(экран − смещение камеры) / масштаб мира**.

Дальше: `worldX` / `worldY` сравниваются с AABB врагов, попадают в `moveTo` / направление тапа и т.д.

## Фаза 4: Камера

1. Обернуть мир и карту в `Container` (`worldRoot`). Введите явную переменную **`worldScale`** (= `worldRoot.scale.x`, при равном масштабе по осям).

2. **Критично:** видимая ширина/высота мира в **мировых пикселях** считаются **только** так (иначе камера «уедет», появятся смещения):

   ```text
   viewW = canvasWidth / worldScale
   viewH = canvasHeight / worldScale
   ```

   Не смешивать: если где-то используете `scale` из другого источника или забываете делить — формулы clamp перестанут соответствовать реальному кадру.

3. Размер карты в пикселях мира: `mapWidthPx = mapWidthTiles * tileWidth`, `mapHeightPx` аналогично.

4. Цель: игрок около центра вьюпорта. Смещение мира (камера), если камера реализована как `worldRoot.x = ...`:

   - Использовать **`viewW` / `viewH`** из п. 2 при **каждом** пересчёте камеры и при **resize**.
   - Тогда ось X:

   ```text
   desiredLeft = player.x - viewW / 2
   camX = clamp(desiredLeft, 0, mapWidthPx - viewW)
   ```

   Аналогично по Y: `camY = clamp(player.y - viewH / 2, 0, mapHeightPx - viewH)`.

   Если `mapWidthPx < viewW`, центрируйте карту (часто `camX = (mapWidthPx - viewW) / 2` или 0 по политике letterbox).

5. Учитывать **resize** canvas / TWA viewport и смену **`worldScale`**: каждый раз заново `viewW` = `canvasWidth / worldScale`, `viewH` = `canvasHeight / worldScale`.

**Критерий:** при движении игрок остаётся в кадре, за края карты камера не уезжает; при смене масштаба и ориентации поведение предсказуемо.

## Фаза 5: Враг, HP, атака по клику

1. Сущность врага: `Enemy`, `Health`, `RenderRef`, `Hitbox`; красный квадрат через реестр + `RenderSystem`.
2. Ввод: `pointertap` → **`screenToWorld`** (см. фазу 3, «Screen → world») → в **`PlayerIntent`**: приоритетно hit-test врага по AABB в **мировых px** → `attackTarget: eid`, иначе обновить поля движения (по правилам desktop/mobile из фазы 3).

3. **Система разрешения атаки** — урон наносится **только** если выполнены **все** условия (иначе «магическая атака из любой точки»):

   - **`attackTarget`** указывает на живую сущность с `Enemy` и без `Dead`;
   - **дистанция** от игрока до цели (центр или ближайшая точка хитбокса) **\< `attackRange`**;
   - **кулдаун** истёк (например **~300 ms** между успешными атаками — зафиксировать константой);
   - опционально: линия видимости без обязательства для MVP квадратов.

   При успехе: `Health[target] -= damage`, старт кулдауна, эмит **`ATTACK`** в протокол-заглушку. При провале любого пункта — не наносить урон, можно обработать только намерение движения.

4. Состояние кулдауна: компонент `AttackCooldown` или поле в `world` / у сущности игрока.

**Критерий:** несколько **валидных** ударов в радиусе с паузой убивают врага; издалека урон не проходит.

## Фаза 6: Смерть и лут

1. Система **death**: для сущностей с `Enemy` и `Health <= 0` — пометить `Dead`, убрать из запросов коллизий/атаки; **`RenderSystem` / реестр** уничтожает или прячет узел по `RenderRef` врага.
2. Создать новую сущность с `Loot`, `Position` = позиция врага, жёлтый/зелёный квадрат (визуал только через **`RenderSystem`** / реестр).

3. **Подбор (MVP):** при пересечении AABB игрока и лута — удалить лут, инкремент счётчика или UI «+1 gold».

   **Нюанс:** на мобиле чистое пересечение хитбоксов даёт **случайный подбор** при проходе мимо — может раздражать. На потом: **автоподбор только в меньшем радиусе** вокруг игрока, **тап по луту** для подтверждения, или задержка «стоять на луте» N ms.

**Критерий:** полный цикл: подойти → бить → лут появляется → подобрать.

## Фаза 7: Полировка mobile-first

1. Виртуальные **четыре направления** или **один джойстик** (опционально для MVP — только тап-to-move: короткий рывок в сторону тапа).
2. Отключить контекстное меню и лишний скролл на области игры.
3. Минимальный UI: полоска HP игрока (если враг будет бить в v2 — пока не нужно).
