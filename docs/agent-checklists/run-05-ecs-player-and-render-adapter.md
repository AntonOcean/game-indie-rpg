# Run 05 — ECS: игрок, реестр представлений, RenderSystem

**Фаза плана:** implementation-plan § «Фаза 3» (п. 1–2, часть 3 без игрового движения).

## Цель

Поднять **bitECS** `world`, компоненты **`Position`** (мировые px), **`Player`**, **`RenderRef`**, **`Hitbox`**; **реестр** `Map<renderId, DisplayObject>` вне ECS; **`RenderSystem`** — **единственное** место, где создаётся/двигается графика игрока (например синий квадрат `Graphics`). Игрок стоит в **валидной** стартовой точке на карте (константа), без ввода и без коллизий в этом ране (или только визуальное совпадение — без проверки тайлов).

## Входные условия

Run-04: карта на сцене, известны `tileWidth` / `tileHeight` и размеры мира.

## Задачи (чек-лист)

- [ ] `createWorld`, регистрация компонентов: `Position`, `Player`, `RenderRef { renderId }`, `Hitbox` (размеры в мировых px, договориться: центр = `Position` или угол — и зафиксировать в коде).
- [ ] Файлы **компонентов ECS** не импортируют Pixi (architecture).
- [ ] Реестр представлений + фабрика: создать узел для игрока, записать `renderId` в `RenderRef`.
- [ ] Каждый кадр: **RenderSystem** читает `Position` (и при необходимости видимость) и обновляет позицию узла в реестре.
- [ ] Прикрепить игрока к тому же **`worldRoot`**, что и карта (если контейнер мира уже есть; иначе ввести минимальный контейнер для карты + сущностей — без формул камеры из run-07).
- [ ] Зафиксировать **порядок систем** в комментарии или константе pipeline (дальше будет расширяться).

## Ограничения

- Не реализовывать `PlayerIntent`, клавиатуру, коллизии и камеру (кроме минимального размещения контейнера).
- Не двигать спрайт из «боевой» логики — только RenderSystem.

## Как проверить

1. Запуск: виден **синий квадрат** игрока на карте в ожидаемой точке.
2. Изменение `Position.x/y` в коде (временный хардкод) сдвигает квадрат — значит синхронизация реестра работает.
3. Нет импортов Pixi в модулях компонентов ECS.

## Выход для следующего рана

Рабочий адаптер рендера и сущность игрока для ввода и физики.
