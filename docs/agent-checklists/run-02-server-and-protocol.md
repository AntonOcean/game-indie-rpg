# Run 02 — Сервер статики и заготовка протокола

**Фаза плана:** implementation-plan § «Фаза 0» (сервер, протокол).

## Цель

Минимальный Node-сервер раздаёт собранный клиент из `dist`; отдельный модуль с типами событий **`MOVE`**, **`ATTACK`**, **`USE_ITEM`** и **no-op** отправителем (лог в консоль или пустая функция) — как в architecture.md.

## Входные условия

Завершён **run-01**: клиент собирается в `dist` (или согласованный путь).

## Задачи (чек-лист)

- [ ] Добавить `apps/server` или `server/` с **express** или **fastify** на порту **3000** (или задокументировать другой порт единообразно).
- [ ] `express.static('dist')` (или путь к артефакту Vite после `vite build`).
- [ ] Опционально: простой `GET /health` → JSON `{ "ok": true }` для проверки.
- [ ] Модуль **`shared/`** или `src/protocol/` (как договорено в монорепо): TypeScript-типы / enum для событий игрока и тел сообщений; функция вроде `sendPlayerEvent(...)` — пока **no-op** или `console.debug`.
- [ ] Корневой или серверный `package.json`: скрипт `start:server` (или аналог из implementation-plan).
- [ ] Кратко описать в README репо или в комментарии: в **dev** клиент крутится отдельно (`vite`), сервер — для прода статики и будущего API.

## Ограничения

- Не поднимать WebSocket и игровую логику на сервере.
- Не блокировать клиент, если сервер не запущен (протокол — заглушка).

## Как проверить

1. `npm run build` (клиент) — `dist` появляется.
2. Запуск сервера — статика отдаётся: открыть в браузере корень, клиент грузится.
3. Импорт типов протокола из клиента **не обязателен** в этом ране; если импортируете — сборка без ошибок.

## Выход для следующего рана

Сервер готов к выкладке статики; протокольные типы существуют для подключения из клиента позже.
